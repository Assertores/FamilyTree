# inconsistent api return value
## Symptoms
<!-- Observable signs indicating the presence of the technical debt (e.g., frequent bugs, slow performance). -->
`GetPersonsMatchingPattern()` is returning a `Person` array instead of an `PersonId` array. this is doubling functionality with `GetPerson()`.

## Context
<!-- Detailed background information, including why the debt was incurred (e.g., rushed deadlines, outdated technologies). -->
having made these two APIs at different times and not having thurowly thought about `GetPersonsMatchingPattern()` caused this overlap. this stayed unnoticed, as the API was introduced during the development of the visualisation executable.

## Impact
<!-- How the debt affects system performance, scalability, maintainability, etc. -->
As `GetPersonsMatchingPattern()` returns an array, and `Person` is a quite big struct, unnessesary big amounts of memory need to be allocated and kept alive.

## Cost of Delay
<!-- Consequences of postponing the resolution of the debt. -->
the fix would cause a breaking API change. therefore waiting means more users need to be fixed manualy, or in worsed case the old behaviour needs to be caped compatible.
